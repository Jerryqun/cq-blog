什么是 RAG

RAG（Retrieval-Augmented Generation，检索增强生成）解决一个核心问题：如何让 AI 基于你的私有数据回答问题，而不是凭空编造？

举个例子：你有一堆公司内部文档（产品手册、API 文档、FAQ），想让 AI 助手能准确回答相关问题。直接把所有文档塞进 prompt 不现实（Token 限制 + 成本），RAG 的做法是：

  1. 把文档转成向量存入数据库
  2. 用户提问时，把问题也转成向量
  3. 在数据库中找到最相似的文档片段
  4. 把这些片段作为上下文喂给 LLM 生成回答
用户提问 → 向量化 → 检索相关文档 → 拼接上下文 → LLM 生成回答

技术选型

组件选择理由
嵌入模型OpenAI text-embedding-3-small性价比高，1536 维
向量数据库Pinecone / Supabase pgvectorPinecone 纯向量数据库，pgvector 可复用 Postgres
框架Vercel AI SDK与 Next.js 无缝集成
文本分块自实现灵活控制分块策略

本文选用 Supabase + pgvector，因为对于前端开发者来说，Supabase 提供了完整的数据库 + 向量存储 + 认证，且免费额度够用。

第一步：文本分块

文档不能整篇丢给向量模型，需要先切成合适大小的块。块太大检索不精准，太小丢失上下文。

// lib/chunking.ts

interface Chunk {
  text: string
  index: number
  metadata: {
    source: string
    page?: number
  }
}

/**
 * 按固定大小分块，带重叠
 * 重叠是为了避免在块边界处切断语义
 */
export function chunkText(
  text: string,
  options: {
    chunkSize?: number    // 每块最大字符数，默认 1000
    overlap?: number      // 相邻块重叠字符数，默认 200
    source?: string       // 文档来源
  } = {}
): Chunk[] {
  const { chunkSize = 1000, overlap = 200, source = 'unknown' } = options

  const chunks: Chunk[] = []
  let startIndex = 0
  let chunkIndex = 0

  while (startIndex < text.length) {
    const endIndex = Math.min(startIndex + chunkSize, text.length)
    const chunkText = text.slice(startIndex, endIndex)

    // 尝试在句子边界处切分，避免切断句子
    const refinedEnd = findSentenceBoundary(chunkText, endIndex)

    chunks.push({
      text: chunkText.slice(0, refinedEnd).trim(),
      index: chunkIndex,
      metadata: { source },
    })

    // 下一块从重叠位置开始
    startIndex += refinedEnd - overlap
    chunkIndex++
  }

  return chunks
}

/**
 * 在给定范围内找到最近的句子结束位置
 */
function findSentenceBoundary(text: string, maxEnd: number): number {
  // 优先在。！？.!? 换行处切分
  const sentenceEnders = /[。！？\.!\?\n]/g
  let lastEnd = maxEnd

  for (let i = maxEnd - 1; i > maxEnd - 100 && i > 0; i--) {
    if (sentenceEnders.test(text[i])) {
      lastEnd = i + 1
      break
    }
  }

  return lastEnd
}

TIP

分块大小没有银弹，需要根据文档类型调整。技术文档建议 500-800 字符，叙述性文档可以 1000-1500。实际项目中我对比了不同分块大小对检索准确率的影响，800 字符 + 150 重叠是综合最优。

第二步：生成向量嵌入

// lib/embeddings.ts
import OpenAI from 'openai'

const openai = new OpenAI()

export async function generateEmbedding(text: string): Promise<number[]> {
  const response = await openai.embeddings.create({
    model: 'text-embedding-3-small',
    input: text,
  })

  return response.data[0].embedding
}

// 批量生成嵌入，减少 API 调用次数
export async function generateEmbeddings(texts: string[]): Promise<number[][]> {
  const batchSize = 100 // OpenAI 单次最多 2048 个输入，实际建议 100 个一批
  const embeddings: number[][] = []

  for (let i = 0; i < texts.length; i += batchSize) {
    const batch = texts.slice(i, i + batchSize)
    const response = await openai.embeddings.create({
      model: 'text-embedding-3-small',
      input: batch,
    })

    embeddings.push(...response.data.map((d) => d.embedding))
  }

  return embeddings
}

第三步：存入向量数据库

Supabase 表结构

-- 启用 pgvector 扩展
create extension if not exists vector;

-- 创建文档表
create table documents (
  id uuid primary key default gen_random_uuid(),
  content text not null,
  source text,
  metadata jsonb,
  embedding vector(1536),  -- text-embedding-3-small 的维度
  created_at timestamptz default now()
);

-- 创建向量索引（IVFFlat 适合中等数据量）
create index on documents using ivfflat (embedding vector_cosine_ops)
with (lists = 100);

-- 创建相似度检索函数
create function match_documents(
  query_embedding vector(1536),
  match_count int default 5,
  filter jsonb default '{}'
) returns table (
  id uuid,
  content text,
  source text,
  metadata jsonb,
  similarity float
) as $$
  select
    id,
    content,
    source,
    metadata,
    1 - (embedding <=> query_embedding) as similarity
  from documents
  where metadata @> filter
  order by embedding <=> query_embedding
  limit match_count
$$ language sql;

索引文档

// lib/indexing.ts
import { createClient } from '@supabase/supabase-js'
import { chunkText } from './chunking'
import { generateEmbeddings } from './embeddings'

const supabase = createClient(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_KEY!
)

export async function indexDocument(text: string, source: string) {
  // 1. 分块
  const chunks = chunkText(text, { chunkSize: 800, overlap: 150, source })
  console.log(`文档分为 ${chunks.length} 个块`)

  // 2. 批量生成嵌入
  const embeddings = await generateEmbeddings(chunks.map((c) => c.text))

  // 3. 存入数据库
  const records = chunks.map((chunk, i) => ({
    content: chunk.text,
    source: chunk.metadata.source,
    metadata: { index: chunk.index, ...chunk.metadata },
    embedding: embeddings[i],
  }))

  const { error } = await supabase.from('documents').insert(records)

  if (error) {
    throw new Error(`索引失败: ${error.message}`)
  }

  console.log(`成功索引 ${records.length} 个文档块`)
}

第四步：检索与生成

// lib/rag.ts
import { openai } from '@ai-sdk/openai'
import { streamText } from 'ai'
import { generateEmbedding } from './embeddings'
import { createClient } from '@supabase/supabase-js'

const supabase = createClient(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_KEY!
)

export async function ragQuery(question: string) {
  // 1. 将问题转为向量
  const queryEmbedding = await generateEmbedding(question)

  // 2. 向量检索相关文档
  const { data: relevantDocs, error } = await supabase.rpc('match_documents', {
    query_embedding: queryEmbedding,
    match_count: 5,
  })

  if (error) throw error

  // 3. 过滤相似度过低的结果
  const filtered = relevantDocs.filter(
    (doc: any) => doc.similarity > 0.7 // 阈值根据实际情况调整
  )

  if (filtered.length === 0) {
    return {
      answer: '抱歉，我在知识库中没有找到与您问题相关的内容。',
      sources: [],
    }
  }

  // 4. 拼接上下文
  const context = filtered
    .map((doc: any, i: number) => `[文档${i + 1}] (来源: ${doc.source})\n${doc.content}`)
    .join('\n\n---\n\n')

  // 5. 调用 LLM 生成回答
  const result = await streamText({
    model: openai('gpt-4o'),
    system: `你是一个专业的知识助手。请根据以下检索到的文档内容回答用户问题。

要求：
1. 只根据提供的文档内容回答，不要编造信息
2. 如果文档中没有相关信息，明确告知用户
3. 回答时标注信息来源，如"根据文档1..."
4. 保持回答简洁准确

检索到的文档：
${context}`,
    prompt: question,
  })

  return {
    answer: await result.text,
    sources: filtered.map((doc: any) => ({
      source: doc.source,
      similarity: doc.similarity,
      snippet: doc.content.slice(0, 100) + '...',
    })),
  }
}

流式 RAG 接口

为了让用户尽快看到回答，我们将检索和生成分离——检索完成后立即开始流式输出：

// app/api/rag/route.ts
import { generateEmbedding } from '@/lib/embeddings'
import { openai } from '@ai-sdk/openai'
import { streamText } from 'ai'
import { createClient } from '@supabase/supabase-js'

const supabase = createClient(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_KEY!
)

export async function POST(req: Request) {
  const { question } = await req.json()

  // 1. 向量检索
  const queryEmbedding = await generateEmbedding(question)
  const { data: docs } = await supabase.rpc('match_documents', {
    query_embedding: queryEmbedding,
    match_count: 5,
  })

  const relevantDocs = (docs || []).filter((d: any) => d.similarity > 0.7)

  if (relevantDocs.length === 0) {
    return new Response(
      JSON.stringify({ error: '未找到相关文档' }),
      { status: 404, headers: { 'Content-Type': 'application/json' } }
    )
  }

  // 2. 流式生成
  const context = relevantDocs
    .map((d: any, i: number) => `[文档${i + 1}] ${d.content}`)
    .join('\n\n')

  const result = streamText({
    model: openai('gpt-4o'),
    system: `根据以下文档回答问题。只使用文档中的信息，不要编造。\n\n${context}`,
    prompt: question,
  })

  // 将来源信息放在响应头中
  const response = result.toDataStreamResponse()
  response.headers.set(
    'X-Sources',
    JSON.stringify(relevantDocs.map((d: any) => d.source))
  )

  return response
}

前端界面

'use client'

import { useState } from 'react'
import { useChat } from 'ai/react'

export default function RagChat() {
  const [sources, setSources] = useState<string[]>([])

  const { messages, input, handleInputChange, handleSubmit, isLoading } = useChat({
    api: '/api/rag',
    onResponse: (response) => {
      // 从响应头读取来源信息
      const sourcesHeader = response.headers.get('X-Sources')
      if (sourcesHeader) {
        setSources(JSON.parse(sourcesHeader))
      }
    },
  })

  return (
    <div className="mx-auto max-w-3xl p-4">
      <div className="space-y-4 mb-4">
        {messages.map((m) => (
          <div key={m.id} className={`p-4 rounded-lg ${
            m.role === 'user' ? 'bg-blue-50' : 'bg-gray-50'
          }`}>
            <div className="font-semibold mb-1">
              {m.role === 'user' ? '你' : 'AI 助手'}
            </div>
            <div className="whitespace-pre-wrap">{m.content}</div>
          </div>
        ))}
      </div>

      {/* 显示引用来源 */}
      {sources.length > 0 && (
        <div className="mb-4 p-3 bg-yellow-50 rounded text-sm">
          <div className="font-semibold mb-1">参考来源：</div>
          <ul className="list-disc list-inside text-gray-600">
            {sources.map((s, i) => (
              <li key={i}>{s}</li>
            ))}
          </ul>
        </div>
      )}

      <form onSubmit={handleSubmit} className="flex gap-2">
        <input
          value={input}
          onChange={handleInputChange}
          placeholder="问我任何关于文档的问题..."
          className="flex-1 border rounded-lg px-4 py-2"
        />
        <button
          type="submit"
          disabled={isLoading}
          className="bg-blue-500 text-white px-6 py-2 rounded-lg disabled:opacity-50"
        >
          {isLoading ? '查询中...' : '提问'}
        </button>
      </form>
    </div>
  )
}

优化：混合检索

纯向量检索在处理精确匹配（如产品名称、错误码）时效果不好。实际项目中我采用了关键词检索 + 向量检索的混合策略：

export async function hybridSearch(question: string) {
  // 1. 向量检索（语义相似）
  const queryEmbedding = await generateEmbedding(question)
  const { data: vectorResults } = await supabase.rpc('match_documents', {
    query_embedding: queryEmbedding,
    match_count: 10,
  })

  // 2. 关键词检索（全文搜索）
  const { data: keywordResults } = await supabase
    .from('documents')
    .select('id, content, source, ts_rank_cd(documents_fts, plainto_tsquery($1)) as rank')
    .textSearch('documents_fts', question)
    .order('rank', { ascending: false })
    .limit(10)

  // 3. 融合排序（RRF - Reciprocal Rank Fusion）
  const rrfK = 60
  const scores: Record<string, number> = {}

  const combineRank = (results: any[], key: string) => {
    results.forEach((item, index) => {
      const id = item[key] || item.id
      scores[id] = (scores[id] || 0) + 1 / (rrfK + index + 1)
    })
  }

  combineRank(vectorResults, 'id')
  combineRank(keywordResults, 'id')

  // 按融合分数排序
  const sortedIds = Object.entries(scores)
    .sort(([, a], [, b]) => b - a)
    .map(([id]) => id)

  // 返回前5个结果
  return sortedIds.slice(0, 5).map((id) => {
    const vectorDoc = vectorResults?.find((d: any) => d.id === id)
    const keywordDoc = keywordResults?.find((d: any) => d.id === id)
    return vectorDoc || keywordDoc
  })
}

常见问题与解决

1. 检索结果不相关

原因：嵌入模型对短文本效果差，或者相似度阈值设置不当。

解决：对问题做 query expansion（查询扩展）——让 LLM 先把用户的口语化问题改写为更标准的检索 query：

const rewriteResult = await generateText({
  model: openai('gpt-4o-mini'),
  prompt: `将以下用户问题改写为更适合向量检索的查询语句，保留核心语义，去除无关词汇：
用户问题：${question}
改写后的查询：`,
})

const searchQuery = rewriteResult.text

2. 上下文超长

如果检索到的文档太多太长，拼接后可能超出模型上下文窗口。解决方案：

  • 控制检索数量（top-k = 3~5）
  • 对检索结果做摘要压缩
  • 使用支持长上下文的模型

3. 知识更新延迟

文档更新后，向量索引需要重建。建议用增量更新策略：只对新增/修改的文档重新生成嵌入。

总结

RAG 是目前 AI 应用最主流的落地形式。对前端开发者来说，核心掌握以下几点：

  1. 分块策略：大小要适中，带重叠，尽量在语义边界切分
  2. 向量检索：理解余弦相似度，合理设置阈值
  3. 上下文工程：检索结果如何拼接、排序、过滤直接影响回答质量
  4. 混合检索：向量 + 关键词双路召回，用 RRF 融合排序
  5. 流式体验：检索和生成分离，让用户尽快看到响应

这个方案我在实际项目中用于构建了内部知识库问答系统，覆盖了 500+ 篇技术文档，问答准确率达到了 85% 以上。RAG 的魅力在于它是"工程"而非"玄学"——每一步都有明确的优化方向。
