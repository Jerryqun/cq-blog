## 为什么需要 AI 代码审查

在团队开发中，Code Review 面临一个现实矛盾：

- **审查者**：每天要看好几个 PR，时间紧、容易敷衍，尤其是大 diff 时倾向于"LGTM"
- **提交者**：希望尽快合并，反复修改等待审查意见很痛苦

AI 代码审查不是要替代人工，而是**做第一道筛子**——自动发现明显的 bug、风格问题、安全隐患，让人工审查聚焦在业务逻辑和架构设计上。

我在团队中落地这套方案三个月后，PR 平均合并时间从 8 小时缩短到 3 小时，人工审查的评论中"格式问题"类减少了 70%。

## 整体架构

```
PR 创建/更新 → GitHub Actions 触发 → 获取 diff → LLM 审查 → 发布 Review 评论
```

核心思路：

1. 通过 GitHub Actions 监听 Pull Request 事件
2. 获取 PR 的 diff 内容
3. 按"文件 + 代码块"粒度拆分，分别送给 LLM 审查
4. 将审查结果作为行内评论发布到 PR 上

## 实现步骤

### 1. GitHub Actions 工作流

```yaml
# .github/workflows/ai-review.yml
name: AI Code Review

on:
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  review:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
      contents: read
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # 获取完整历史，用于 diff

      - name: Run AI Review
        uses: actions/github-script@v7
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
        with:
          script: |
            const script = require('./scripts/ai-review.cjs')
            await script({ github, context, core })
```

### 2. 获取 PR Diff

```javascript
// scripts/ai-review.cjs
const { execSync } = require('child_process')

async function getDiffFiles(baseSha, headSha) {
  // 获取 diff，带行号信息
  const diff = execSync(
    `git diff ${baseSha}...${headSha} --unified=5 --diff-filter=ACMR`,
    { encoding: 'utf-8', maxBuffer: 1024 * 1024 * 10 }
  )

  // 解析 diff 为按文件分组的代码块
  return parseDiff(diff)
}

function parseDiff(diffText) {
  const files = []
  const diffLines = diffText.split('\n')

  let currentFile = null
  let currentHunk = null
  let oldLine = 0
  let newLine = 0

  for (const line of diffLines) {
    // 匹配文件头
    const fileMatch = line.match(/^diff --git a\/(.+?) b\/(.+)$/)
    if (fileMatch) {
      currentFile = {
        path: fileMatch[2],
        hunks: [],
      }
      files.push(currentFile)
      continue
    }

    // 匹配 hunk 头：@@ -oldStart,oldLen +newStart,newLen @@
    const hunkMatch = line.match(/^@@ -(\d+)(?:,\d+)? \+(\d+)(?:,\d+)? @@/)
    if (hunkMatch) {
      oldLine = parseInt(hunkMatch[1])
      newLine = parseInt(hunkMatch[2])
      currentHunk = {
        oldStart: oldLine,
        newStart: newLine,
        lines: [],
      }
      currentFile.hunks.push(currentHunk)
      continue
    }

    if (currentHunk) {
      if (line.startsWith('+') && !line.startsWith('+++')) {
        // 新增的行
        currentHunk.lines.push({
          type: 'add',
          content: line.slice(1),
          lineNumber: newLine,
        })
        newLine++
      } else if (line.startsWith('-') && !line.startsWith('---')) {
        // 删除的行
        currentHunk.lines.push({
          type: 'remove',
          content: line.slice(1),
          lineNumber: oldLine,
        })
        oldLine++
      } else {
        // 上下文行
        currentHunk.lines.push({
          type: 'context',
          content: line.startsWith(' ') ? line.slice(1) : line,
          lineNumber: newLine,
        })
        oldLine++
        newLine++
      }
    }
  }

  return files
}

module.exports = { getDiffFiles }
```

### 3. LLM 审查逻辑

```javascript
const { OpenAI } = require('openai')
const openai = new OpenAI()

const REVIEW_SYSTEM_PROMPT = `你是一个严格的代码审查专家。审查以下代码变更，找出潜在问题。

## 审查维度
1. **Bug 风险**：空指针、未处理异常、竞态条件、边界错误
2. **安全漏洞**：XSS、注入攻击、硬编码密钥、不安全的数据处理
3. **性能问题**：不必要的渲染、内存泄漏、N+1 查询
4. **代码质量**：命名不规范、重复代码、缺少类型定义、过度复杂
5. **最佳实践**：React hooks 使用规范、TypeScript 类型安全

## 输出要求
以 JSON 数组返回发现的问题，每个问题包含：
- "line": 行号（必填，数字）
- "severity": "high" | "medium" | "low"
- "category": 问题类别
- "message": 问题描述（简洁，一句话）
- "suggestion": 修复建议（可选）

如果没有发现问题，返回空数组 []。
只返回 JSON，不要其他文字。`

async function reviewCode(filePath, fileContent, language) {
  // 只审查有实质内容变更的文件
  if (fileContent.length > 8000) {
    // 大文件分块审查
    return reviewLargeFile(filePath, fileContent, language)
  }

  const response = await openai.chat.completions.create({
    model: 'gpt-4o',
    response_format: { type: 'json_object' },
    messages: [
      { role: 'system', content: REVIEW_SYSTEM_PROMPT },
      {
        role: 'user',
        content: `文件: ${filePath}\n语言: ${language}\n\n代码:\n${fileContent}`,
      },
    ],
    temperature: 0.1, // 低温度保证一致性
  })

  try {
    const result = JSON.parse(response.choices[0].message.content)
    return result.issues || result || []
  } catch {
    return []
  }
}
```

### 4. 发布 Review 评论

```javascript
async function publishReview({ github, context }, files, allIssues) {
  // 将问题按文件分组，转为 GitHub Review comments 格式
  const comments = allIssues
    .filter((issue) => issue.line && issue.line > 0)
    .map((issue) => ({
      path: issue.filePath,
      line: issue.line,
      side: 'RIGHT',
      body: formatComment(issue),
    }))

  if (comments.length === 0) {
    // 没有问题，发布 approve
    await github.rest.pulls.createReview({
      ...context.repo,
      pull_number: context.payload.pull_request.number,
      event: 'APPROVE',
      body: '✅ AI 审查通过，未发现问题。',
    })
    return
  }

  // 发布审查评论
  await github.rest.pulls.createReview({
    ...context.repo,
    pull_number: context.payload.pull_request.number,
    event: 'COMMENT',
    body: `🤖 AI 审查完成，发现 ${comments.length} 个问题。`,
    comments,
  })
}

function formatComment(issue) {
  const icon = { high: '🔴', medium: '🟡', low: '🟢' }[issue.severity] || '⚪'
  return `${icon} **${issue.severity.toUpperCase()}** - ${issue.category}

${issue.message}

${issue.suggestion ? `> 💡 **建议**: ${issue.suggestion}` : ''}

<sub>🤖 AI 自动审查</sub>`
}
```

### 5. 完整流程

```javascript
module.exports = async function ({ github, context, core }) {
  const { getDiffFiles, reviewCode, publishReview } = require('./review-utils')

  const pr = context.payload.pull_request
  core.info(`开始审查 PR #${pr.number}: ${pr.title}`)

  // 1. 获取变更文件
  const files = await getDiffFiles(
    context.payload.pull_request.base.sha,
    context.payload.pull_request.head.sha
  )

  // 2. 过滤需要审查的文件
  const reviewableFiles = files.filter((file) => {
    // 只审查代码文件
    const exts = ['.ts', '.tsx', '.js', '.jsx', '.vue', '.py']
    const ext = file.path.substring(file.path.lastIndexOf('.'))
    if (!exts.includes(ext)) return false

    // 跳过自动生成文件
    if (file.path.includes('node_modules') || file.path.endsWith('.min.js')) {
      return false
    }

    // 跳过只有删除的文件
    const hasAdditions = file.hunks.some((hunk) =>
      hunk.lines.some((l) => l.type === 'add')
    )

    return hasAdditions
  })

  core.info(`共 ${files.length} 个文件变更，${reviewableFiles.length} 个需要审查`)

  // 3. 逐文件审查
  const allIssues = []
  for (const file of reviewableFiles) {
    // 构建带行号的代码内容
    const codeContent = file.hunks
      .map((hunk) =>
        hunk.lines
          .filter((l) => l.type === 'add' || l.type === 'context')
          .map((l) => `${l.lineNumber}: ${l.content}`)
          .join('\n')
      )
      .join('\n...\n')

    const issues = await reviewCode(file.path, codeContent, detectLanguage(file.path))

    // 为每个问题附加文件路径
    issues.forEach((issue) => {
      issue.filePath = file.path
    })

    allIssues.push(...issues)

    core.info(`${file.path}: 发现 ${issues.length} 个问题`)
  }

  // 4. 过滤误报
  const filtered = filterFalsePositives(allIssues)

  // 5. 发布评论
  await publishReview({ github, context }, files, filtered)

  core.info(`审查完成，共发布 ${filtered.length} 条评论`)
}

function filterFalsePositives(issues) {
  // 去重：同一行同一类问题只保留一个
  const seen = new Set()
  return issues.filter((issue) => {
    const key = `${issue.filePath}:${issue.line}:${issue.category}`
    if (seen.has(key)) return false
    seen.add(key)
    return true
  })
}

function detectLanguage(filePath) {
  const ext = filePath.substring(filePath.lastIndexOf('.'))
  const map = {
    '.ts': 'TypeScript', '.tsx': 'TypeScript/React',
    '.js': 'JavaScript', '.jsx': 'JavaScript/React',
    '.vue': 'Vue', '.py': 'Python',
  }
  return map[ext] || 'Unknown'
}
```

## 进阶优化

### 按代码块审查而非整文件

大文件整文件审查容易超出 Token 限制，也容易遗漏。按 hunk 粒度审查更精准：

```javascript
async function reviewByHunks(filePath, hunks, language) {
  const allIssues = []

  for (const hunk of hunks) {
    // 只审查新增的代码
    const addedLines = hunk.lines.filter((l) => l.type === 'add')
    if (addedLines.length === 0) continue

    // 带上下文构建审查内容
    const contextLines = hunk.lines.map((l) => {
      const prefix = l.type === 'add' ? '+' : l.type === 'remove' ? '-' : ' '
      return `${prefix} ${l.lineNumber || ''} | ${l.content}`
    }).join('\n')

    const issues = await reviewCode(
      filePath,
      contextLines,
      language
    )

    allIssues.push(...issues)
  }

  return allIssues
}
```

### 评论去重：避免重复审查

PR 更新（push 新 commit）时会重新触发审查，需要避免对已审查过的代码重复评论：

```javascript
async function getExistingComments(github, context) {
  const { data: reviews } = await github.rest.pulls.listReviews({
    ...context.repo,
    pull_number: context.payload.pull_request.number,
  })

  const comments = []
  for (const review of reviews) {
    if (review.user.login === 'github-actions[bot]') {
      const { data: reviewComments } = await github.rest.pulls.listCommentsForReview({
        ...context.repo,
        pull_number: context.payload.pull_request.number,
        review_id: review.id,
      })
      comments.push(...reviewComments)
    }
  }

  return comments
}

function filterAlreadyCommented(newIssues, existingComments) {
  const commentedSet = new Set(
    existingComments.map((c) => `${c.path}:${c.line}`)
  )

  return newIssues.filter(
    (issue) => !commentedSet.has(`${issue.filePath}:${issue.line}`)
  )
}
```

### 自定义审查规则

不同团队的规范不同，需要支持自定义规则。把规则作为 prompt 的一部分：

```javascript
// 审查规则配置文件
const REVIEW_RULES = {
  // 通用规则
  general: [
    '不允许使用 any 类型',
    '函数复杂度不超过 15',
    '所有公开 API 必须有 JSDoc 注释',
  ],

  // React 专项规则
  react: [
    'useEffect 必须有完整的依赖数组',
    '禁止在 useEffect 中直接使用 async',
    '组件 Props 必须定义 interface',
    '事件监听器必须在 useEffect 中清理',
  ],

  // 安全专项规则
  security: [
    '禁止使用 dangerouslySetInnerHTML',
    '用户输入必须做 XSS 过滤',
    '禁止在前端代码中硬编码 API Key',
  ],
}

function buildReviewPrompt(rules) {
  const allRules = [
    ...REVIEW_RULES.general,
    ...REVIEW_RULES.react,
    ...REVIEW_RULES.security,
    ...rules, // 项目自定义规则
  ]

  return `## 项目特定审查规则
${allRules.map((r, i) => `${i + 1}. ${r}`).join('\n')}
`
}
```

在项目根目录放一个 `.ai-review.json` 配置文件：

### 误报处理

AI 审查最大的问题是误报。我们的处理策略：

1. **严重度分级**：只有 high 级别阻止合并，medium/low 仅提示
2. **反馈机制**：开发者可以回复 `false-positive` 标记误报，用于后续优化 prompt
3. **白名单**：对特定文件路径或模式跳过审查

```javascript
function shouldBlockMerge(issues) {
  return issues.some(
    (i) => i.severity === 'high' && i.category !== 'style'
  )
}
```

### 与人工审查协同

AI 审查的定位是"第一道防线"，处理机械性的、模式化的问题。人工审查仍然不可替代，聚焦在：

- 业务逻辑正确性
- 架构设计合理性
- 代码可维护性
- 边界场景考虑

我们的流程是：PR 提交 → AI 自动审查（5 分钟内完成）→ 人工审查。AI 的评论帮助审查者快速定位风险点，提高审查效率。

## 总结

AI 代码审查是一个投入产出比很高的 AI 落地场景。核心要点：

1. **diff 解析是基础**：精确解析行号，让评论能定位到具体代码行
2. **分块审查**：按 hunk 粒度审查，避免 Token 超限，提高精准度
3. **去重与过滤**：避免重复评论，过滤误报，尊重已有审查结果
4. **可配置规则**：支持项目自定义规则，适应不同团队规范
5. **与人工协同**：AI 做第一道筛子，人工聚焦业务和架构

这套方案最适合的场景是：团队有一定规模（5 人以上），PR 频率较高，代码规范需要统一。如果你团队还在用 ESLint + Prettier 做代码检查，AI 审查是对它们的智能补充——能理解上下文语义的审查，是静态规则无法企及的。
